On the Firestore side, five collections cover everything you listed. `rooms` is the catalog — each doc is a bookable product (slug, title, type, price, images, capacity/availability) and `src/rooms.html` paginates over `src/_data/rooms.json` at build time to generate one detail page per item, while live availability is still checked against Firestore at request time. `offers` references a `roomId`, carries discount info and a validity window, and is what renders as the offer cards on the homepage and the packages/offers page. `reservations` is the central booking record — room reference, user (or guest) contact, dates, status (`pending` → `confirmed` → `completed`/`cancelled`), and a price snapshot. `notifications` is a per-user feed written by Cloud Functions in response to reservation events, read live via `onSnapshot`. `users` ties Firebase Auth uids to a role (`customer`/`admin`) so the dashboard and security rules can branch on it.

The flow ties together like this: a visitor browses `rooms.html`, clicks into a product room page, `room-page.js` shows live availability plus any matching `offers` badge. Submitting the reservation modal calls a callable function in `routes/booking.js`, which validates the payload against `schema_reservation.json`, checks availability, and writes to `reservations`. That write triggers `onReservationCreate`, which (via `notify.js`) drops a doc into `notifications` for both the customer and an admin feed — `notifications.js` on the client and `reservations-admin.js` in the dashboard both pick it up in real time. Using Firebase Auth with anonymous sign-in for guests means even non-account visitors get a notification thread without forcing a login wall.

For deployment, the standard pairing is GitHub Actions + Firebase Hosting: the merge workflow runs `npx @11ty/eleventy` then deploys `public/` via the Firebase GitHub Action (service account JSON as a repo secret), with a separate step or workflow for `firebase deploy --only functions,firestore` since those change less often. One more thing worth flagging now: Firebase web config (apiKey, projectId, etc.) isn't actually secret — it's fine in client JS, protected by `firestore.rules` instead — but anything `notify.js` needs for email (SMTP/SendGrid keys) should go through `firebase functions:secrets:set`, not `.env`.

The third piece — HR — is the part that changes the shape of everything else, so let's start there conceptually and work outward. Right now `users` only needs to describe customers. The moment you add employees, `users` becomes a shared identity table for *anyone with a login*, and `role` becomes the field that decides which extension document — `customers/{uid}` or `employees/{employeeId}` — actually applies. Everything below (the admin directory, the Storage layout, the HR collections) hangs off that one decision.

One foundational point before the trees: with Eleventy + Firebase Hosting, every file under `public/` is publicly servable HTML — there's no server-side route guard the way there would be with an Express admin panel. So `src/admin/**` pages are just UI shells (markup + a script tag). The real access boundary is Firebase Auth custom claims + Firestore/Storage security rules, checked when the page's JS talks to the database. That's worth keeping in mind especially for payroll and Iqama data.
